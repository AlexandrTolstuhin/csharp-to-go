Сборка мусора (GC)

Введение

💡 Рекомендуется: Перед изучением этого раздела ознакомьтесь с 2.2a Аллокатор памяти для понимания низкоуровневых структур (mheap, mspan, mcache), которые GC использует для работы.

Сборка мусора в Go разработана с фокусом на низкие паузы (< 1ms) и предсказуемость, что критично для высоконагруженных сервисов. Философия Go GC радикально отличается от .NET.

💡 Для C# разработчиков: В .NET GC оптимизирован для throughput (пропускной способности) с паузами до 100ms+. В Go GC оптимизирован для low latency с паузами < 1ms.

🚀 Актуально (2026): Этот раздел описывает GC в Go 1.26 (текущая стабильная версия). Go 1.26 включает Green Tea GC по умолчанию — новый алгоритм сканирования со снижением CPU затрат на GC на 10-40% за счёт улучшенной cache-locality. Go 1.24 добавил Swiss Tables для встроенных map и пакет weak для слабых ссылок. Существующий код получает эти улучшения без изменений.

Архитектура GC в Go vs .NET

.NET GC (для контекста)

C# (Generational GC):

// .NET GC работает с поколениями
// Gen0: короткоживущие объекты (< 1MB обычно)
// Gen1: средней продолжительности
// Gen2: долгоживущие объекты + LOH (Large Object Heap)

var list = new List<int>(); // Аллокация в Gen0

// После нескольких Gen0 сборок выживший объект переходит в Gen1
// Затем в Gen2

// Полная сборка (Gen2 + LOH) может занять 100ms+
GC.Collect(2, GCCollectionMode.Forced); // Принудительная сборка

Проблемы .NET GC:

  • Stop-the-World паузы при полной сборке Gen2
  • Unpredictable latency — сложно предсказать, когда будет пауза
  • Large Object Heap не компактируется → fragmentation

Go GC (Tri-Color Mark-and-Sweep)

Go (Concurrent Mark-Sweep):

// Go GC не использует поколения
// Все объекты обрабатываются одинаково
// Concurrent сборка с минимальными паузами

package main

import (
    "fmt"
    "runtime"
    "runtime/debug"
)

func main() {
    // Получение статистики GC
    var m runtime.MemStats
    runtime.ReadMemStats(&m)

    fmt.Printf("Аллокировано: %d MB\n", m.Alloc/1024/1024)
    fmt.Printf("Всего аллокаций: %d MB\n", m.TotalAlloc/1024/1024)
    fmt.Printf("Системная память: %d MB\n", m.Sys/1024/1024)
    fmt.Printf("Количество GC: %d\n", m.NumGC)
    fmt.Printf("Последняя пауза GC: %v\n", m.PauseNs[(m.NumGC+255)%256])

    // Настройка GC
    debug.SetGCPercent(100) // GOGC=100 (по умолчанию)

    // Принудительный запуск GC (редко нужно в продакшене)
    runtime.GC()
}

Преимущества Go GC:

  • ✅ Predictable latency — паузы < 1ms
  • ✅ Concurrent — работает параллельно с приложением
  • ✅ Простая настройка — два основных параметра

Недостатки:

  • ❌ Нет компактификации — возможна фрагментация памяти
  • ❌ Больше CPU overhead — из-за concurrent работы
  • ❌ Нет оптимизации для короткоживущих объектов (как Gen0 в .NET)

Tri-Color Mark-and-Sweep алгоритм

Как работает

Go GC использует три цвета для маркировки объектов:

┌─────────┐
│  White  │ ← Потенциально мертвые объекты (будут удалены)
└─────────┘
┌─────────┐
│  Grey   │ ← Живые объекты, которые нужно просканировать
└─────────┘
┌─────────┐
│  Black  │ ← Живые объекты, уже просканированные
└─────────┘

Фазы GC

// Псевдокод работы Go GC

// 1. SWEEP TERMINATION (STW ~ 10-30μs)
//    - Завершение предыдущей фазы sweep
//    - Включение write barriers

// 2. MARK (concurrent)
//    - Все объекты изначально WHITE
//    - Корневые объекты → GREY
//    - Сканирование GREY → BLACK, их потомки → GREY
//    - Работает параллельно с приложением (25% CPU)

// 3. MARK TERMINATION (STW ~ 60-90μs)
//    - Завершение маркировки
//    - Выключение write barriers
//    - Подготовка к sweep

// 4. SWEEP (concurrent)
//    - Удаление WHITE объектов
//    - Возврат памяти в аллокатор

Сравнение с .NET

C# (Mark-Compact):

// .NET GC Gen0 сборка:
// 1. Stop-the-World
// 2. Mark живых объектов
// 3. Compact (перемещение объектов в начало кучи)
// 4. Обновление указателей
// 5. Resume

// Пауза: 1-5ms для Gen0, 10-100ms+ для Gen2

Go (Mark-Sweep):

// Go GC:
// 1. STW (10-30μs) - sweep termination
// 2. Concurrent Mark (работает параллельно)
// 3. STW (60-90μs) - mark termination
// 4. Concurrent Sweep

// Общая STW пауза: < 1ms

Визуализация

.NET GC (Generational):
┌────────────────────────────────┐
│  Gen0  │  Gen1  │     Gen2     │
│ (fast) │ (mid)  │    (slow)    │
└────────────────────────────────┘
    ↓         ↓          ↓
  1-5ms    5-20ms    10-100ms+

Go GC (Concurrent):
┌────────────────────────────────┐
│      Concurrent Mark/Sweep     │
│     (работает параллельно)     │
└────────────────────────────────┘
STW: 10-30μs + 60-90μs = < 1ms

Улучшения GC в Go 1.23-1.26

🚀 Go 1.23 (август 2024): Автоматическая сборка Timer/Ticker — исправлена критическая утечка памяти. Теперь неостановленные таймеры собираются GC автоматически.

Swiss Tables для встроенных map (Go 1.24)

Go 1.24 переписал внутреннюю реализацию встроенных map с использованием Swiss Tables — алгоритма Google с SIMD-оптимизациями:

  • ~2-3% снижение CPU на map-тяжёлых workload-ах
  • Лучшая cache-locality: ключи группируются по 8 в «bucket» с probe sequence
  • SIMD-инструкции на Ice Lake, AMD Zen 4+ для параллельного поиска по bucket
  • Меньше GC-давления за счёт более компактного расположения данных
// API не изменился — код работает быстрее «бесплатно»
m := make(map[string]int)
m["key"] = 42

Пакет weak.Pointer[T] (Go 1.24)

Go 1.24 добавил пакет weak для слабых ссылок — указателей, которые не препятствуют сборке мусора. Аналог WeakReference<T> в C#.

C#:

// WeakReference<T> — не предотвращает GC
var weakRef = new WeakReference<ExpensiveObject>(obj);
if (weakRef.TryGetTarget(out var target))
{
    target.DoWork();
}

Go 1.24 (weak.Pointer[T]):

import "weak"

// Создаём слабую ссылку
obj := &ExpensiveObject{Data: "valuable"}
weakPtr := weak.Make(obj)

// Объект жив — можно разыменовать
if strong := weakPtr.Value(); strong != nil {
    strong.DoWork()
}

// После GC: weakPtr.Value() вернёт nil
obj = nil
runtime.GC() // Объект может быть собран

Применение weak.Pointer[T]:

// Кэш без утечек памяти (замена finalizer-паттернов)
type Cache struct {
    entries map[string]weak.Pointer[CacheEntry]
}

func (c *Cache) Get(key string) *CacheEntry {
    if wp, ok := c.entries[key]; ok {
        return wp.Value() // nil если GC собрал объект
    }
    return nil
}

func (c *Cache) Set(key string, entry *CacheEntry) {
    c.entries[key] = weak.Make(entry)
}

💡 Для C# разработчиков: weak.Pointer[T] — типобезопасная версия WeakReference<T>. Используйте для кэшей и наблюдателей, где нежелательно удерживать объекты в памяти.

Green Tea GC (Go 1.26 — по умолчанию)

🚀 Go 1.26 (февраль 2026): Green Tea GC стал алгоритмом по умолчанию.

Что изменилось:

  • Переход с per-object на per-span сканирование
  • GC сканирует целые mspan блоки, а не отдельные объекты — меньше cache misses
  • 10-40% снижение CPU затрат на GC в типичных сценариях
  • Дополнительно ~10% ускорение на процессорах с SIMD (Ice Lake, AMD Zen 4+) за счёт параллельной обработки span-bitmap
  • Никаких изменений в коде не требуется — улучшение прозрачное
// Пример: сервис с высокой аллокацией — выигрыш от Green Tea GC
func processRequests(requests []*Request) {
    for _, req := range requests {
        // Каждая обработка создаёт временные объекты
        result := &Result{
            Data:   transform(req.Data),
            Meta:   extractMeta(req),
        }
        send(result)
    }
    // Green Tea GC: сканирует mspan целиком — меньше cache misses
    // при очистке временных result/meta объектов
}

Write Barriers и concurrent GC

Проблема concurrent GC

Когда GC работает concurrent (параллельно с приложением), возникает проблема:

// Проблема: приложение изменяет граф объектов во время маркировки

// До GC:
// A (black) → B (white)

// Во время GC приложение делает:
// A.field = C  // C был white, не отмаркирован
// B.field = nil

// Результат: C может быть удален, хотя он живой!

Write Barriers

Write barrier — это код, который вставляется компилятором при записи указателей:

// Псевдокод write barrier в Go

func writePointer(slot *unsafe.Pointer, ptr unsafe.Pointer) {
    // Если GC активен и объект black пытается указать на white
    if gcActive && isBlack(slot) && isWhite(ptr) {
        // Закрасить white объект в grey
        shade(ptr)
    }
    *slot = ptr // Реальная запись
}

Стоимость write barriers:

// Write barriers добавляют overhead к записям указателей
// Но работают только во время GC (не всегда)

// Примерная стоимость:
// Без GC: 1-2 CPU цикла на запись указателя
// С GC:   10-20 CPU циклов на запись указателя

// Оптимизация: избегайте частых записей указателей в hot path

GOGC и GOMEMLIMIT

GOGC (главный параметр)

GOGC определяет, когда запускать следующий GC:

# GOGC=100 (по умолчанию)
# Следующий GC запускается, когда heap вырастет на 100% от live set

# Пример:
# Live set = 100 MB (объекты, которые пережили предыдущий GC)
# GOGC=100 → следующий GC при heap = 200 MB
# GOGC=200 → следующий GC при heap = 300 MB
# GOGC=50  → следующий GC при heap = 150 MB

Go пример:

package main

import (
    "fmt"
    "runtime"
    "runtime/debug"
)

func demonstrateGOGC() {
    // Установка GOGC программно
    oldGOGC := debug.SetGCPercent(200) // Удвоить интервал между GC
    fmt.Printf("Старый GOGC: %d\n", oldGOGC)

    // Или через переменную окружения:
    // GOGC=200 go run main.go

    // Отключить GC (используется редко, для тестов)
    debug.SetGCPercent(-1)

    // Вернуть обратно
    debug.SetGCPercent(100)
}

func demonstrateMemoryTrade() {
    // Trade-off: больше GOGC → больше памяти, меньше CPU на GC

    // GOGC=100 (по умолчанию):
    // - Средний расход памяти
    // - Средний CPU на GC (~1-5%)

    // GOGC=200:
    // - Вдвое больше памяти
    // - Вдвое меньше CPU на GC

    // GOGC=50:
    // - Меньше памяти
    // - Больше CPU на GC

    var m runtime.MemStats
    runtime.ReadMemStats(&m)

    fmt.Printf("Live heap: %d MB\n", m.Alloc/1024/1024)
    fmt.Printf("Sys memory: %d MB\n", m.Sys/1024/1024)
    fmt.Printf("GC cycles: %d\n", m.NumGC)
}

Жесткий лимит памяти (аналог GC.MemoryLimit в .NET):

Escape Analysis: Stack vs Heap

Что такое Escape Analysis

Компилятор Go анализирует, где разместить переменную:

  • Stack — быстро, автоматически очищается при возврате из функции
  • Heap — медленно, требует GC для очистки

💡 Для C# разработчиков: В C# все reference types всегда в куче. В Go компилятор сам решает, где разместить объект, даже если это указатель.

Правила Escape Analysis

package main

import "fmt"

// ✅ НЕ escape: возвращается значение
func noEscape1() int {
    x := 42
    return x // x на стеке
}

// ✅ НЕ escape: локальное использование
func noEscape2() {
    s := make([]int, 100) // s на стеке (если не слишком большой)
    _ = s
}

// ❌ ESCAPE: возвращается указатель
func escape1() *int {
    x := 42
    return &x // x escape в heap (возвращается наружу)
}

// ❌ ESCAPE: сохраняется в глобальную переменную
var global *int

func escape2() {
    x := 42
    global = &x // x escape в heap
}

// ❌ ESCAPE: передается в горутину
func escape3() {
    x := 42
    go func() {
        fmt.Println(x) // x escape в heap (используется в другой горутине)
    }()
}

// ❌ ESCAPE: слишком большой для стека
func escape4() {
    // Массивы > ~64KB обычно escape в heap
    var huge [100000]int
    _ = huge // escape в heap
}

// ❌ ESCAPE: передается в интерфейс
func escape5() {
    x := 42
    fmt.Println(x) // x escape в heap (fmt.Println принимает interface{})
}

Проверка escape analysis

# Компиляция с выводом escape analysis
go build -gcflags="-m -m" main.go

# Пример вывода:
# ./main.go:10:2: x escapes to heap
# ./main.go:10:2:   flow: ~r0 = &x
# ./main.go:9:6: moved to heap: x

Go пример с анализом:

package main

import "fmt"

func stackAllocation() {
    // Анализ покажет: "does not escape"
    x := 42
    y := x * 2
    _ = y
}

func heapAllocation() *int {
    // Анализ покажет: "moved to heap: x"
    x := 42
    return &x
}

func interfaceEscape() {
    // Анализ покажет: "x escapes to heap"
    x := 42
    fmt.Println(x) // interface{} вызывает escape
}

func main() {
    stackAllocation()  // Быстро, без GC
    _ = heapAllocation() // Медленно, требует GC
    interfaceEscape()   // Escape из-за interface{}
}

Профилирование и диагностика

runtime.MemStats

package main

import (
    "fmt"
    "runtime"
    "time"
)

func printMemStats() {
    var m runtime.MemStats
    runtime.ReadMemStats(&m)

    fmt.Printf("=== Memory Stats ===\n")
    fmt.Printf("Alloc      = %v MB\n", bToMb(m.Alloc))
    fmt.Printf("TotalAlloc = %v MB\n", bToMb(m.TotalAlloc))
    fmt.Printf("Sys        = %v MB\n", bToMb(m.Sys))
    fmt.Printf("NumGC      = %v\n", m.NumGC)
    fmt.Printf("HeapAlloc  = %v MB\n", bToMb(m.HeapAlloc))
    fmt.Printf("HeapSys    = %v MB\n", bToMb(m.HeapSys))
    fmt.Printf("HeapIdle   = %v MB\n", bToMb(m.HeapIdle))
    fmt.Printf("HeapInuse  = %v MB\n", bToMb(m.HeapInuse))
    fmt.Printf("HeapReleased = %v MB\n", bToMb(m.HeapReleased))

    // Паузы GC
    fmt.Printf("PauseTotal = %v ms\n", float64(m.PauseTotalNs)/1e6)
    fmt.Printf("Last Pause = %v μs\n", m.PauseNs[(m.NumGC+255)%256]/1000)
}

func bToMb(b uint64) uint64 {
    return b / 1024 / 1024
}

func main() {
    printMemStats()

    // Создаем нагрузку
    data := make([][]byte, 0)
    for i := 0; i < 100; i++ {
        data = append(data, make([]byte, 1024*1024)) // 1 MB
    }

    time.Sleep(100 * time.Millisecond)
    printMemStats()

    // Освобождаем
    data = nil
    runtime.GC()
    time.Sleep(100 * time.Millisecond)
    printMemStats()
}

package main

import (
    "fmt"
    "strings"
)

// ❌ ПЛОХО: много маленьких аллокаций
func badStringConcat(items []string) string {
    result := ""
    for _, item := range items {
        result += item // Каждая операция = новая аллокация
    }
    return result
}

// ✅ ХОРОШО: strings.Builder (минимум аллокаций)
func goodStringConcat(items []string) string {
    var sb strings.Builder
    sb.Grow(len(items) * 10) // Предварительная аллокация
    for _, item := range items {
        sb.WriteString(item)
    }
    return sb.String()
}

// ❌ ПЛОХО: аллокация на каждой итерации
func badLoop() {
    for i := 0; i < 1000; i++ {
        data := make([]int, 100) // 1000 аллокаций
        _ = data
    }
}

// ✅ ХОРОШО: переиспользование буфера
func goodLoop() {
    data := make([]int, 100) // Одна аллокация
    for i := 0; i < 1000; i++ {
        // Очистка и переиспользование
        for j := range data {
            data[j] = 0
        }
        _ = data
    }
}

// ❌ ПЛОХО: возврат указателя на локальную переменную
func badReturn() *User {
    u := User{Name: "Alice"} // Escape в heap
    return &u
}

// ✅ ХОРОШО: возврат значения (если возможно)
func goodReturn() User {
    return User{Name: "Alice"} // На стеке (если не escape)
}

Batch обработка

package main

// ❌ ПЛОХО: обработка по одному
func badBatch(items []Item) {
    for _, item := range items {
        process(item) // Много syscalls / аллокаций
    }
}

// ✅ ХОРОШО: batch обработка
func goodBatch(items []Item) {
    const batchSize = 100
    for i := 0; i < len(items); i += batchSize {
        end := i + batchSize
        if end > len(items) {
            end = len(items)
        }
        processBatch(items[i:end]) // Меньше overhead
    }
}

sync.Pool и переиспользование объектов

Что такое sync.Pool

sync.Pool — это кэш для переиспользования объектов, чтобы снизить давление на GC.

💡 Для C# разработчиков: Аналог ArrayPool<T> в .NET, но более общий.

Базовое использование

package main

import (
    "bytes"
    "sync"
)

var bufferPool = sync.Pool{
    New: func() interface{} {
        // Функция создания нового объекта
        return new(bytes.Buffer)
    },
}

func processData(data []byte) []byte {
    // Получить буфер из пула
    buf := bufferPool.Get().(*bytes.Buffer)
    defer func() {
        buf.Reset() // Очистить перед возвратом
        bufferPool.Put(buf) // Вернуть в пул
    }()

    // Использование буфера
    buf.Write(data)
    buf.WriteString(" processed")

    // Копирование результата (buf вернется в пул)
    result := make([]byte, buf.Len())
    copy(result, buf.Bytes())
    return result
}

func main() {
    data := []byte("hello")
    result := processData(data)
    println(string(result))
}

sync.Pool для структур

package main

import (
    "sync"
)

type Request struct {
    ID   int
    Data []byte
}

var requestPool = sync.Pool{
    New: func() interface{} {
        return &Request{
            Data: make([]byte, 0, 1024), // Предварительная аллокация
        }
    },
}

func handleRequest(id int, data []byte) {
    // Получить объект из пула
    req := requestPool.Get().(*Request)
    defer requestPool.Put(req)

    // Инициализация
    req.ID = id
    req.Data = append(req.Data[:0], data...) // Переиспользование слайса

    // Обработка
    processRequest(req)

    // НЕ нужно очищать вручную - это сделает New при необходимости
}

func processRequest(req *Request) {
    // Ваша логика
}

Сравнение с C#

using System.Buffers;

public class Example
{
    private static ArrayPool<byte> pool = ArrayPool<byte>.Shared;

    public void ProcessData(byte[] data)
    {
        // Аренда массива из пула
        byte[] buffer = pool.Rent(1024);
        try
        {
            // Использование buffer
            Array.Copy(data, buffer, data.Length);
            // ...
        }
        finally
        {
            // Возврат в пул
            pool.Return(buffer, clearArray: true);
        }
    }
}

Когда использовать sync.Pool

✅ Используйте sync.Pool:

  • Временные буферы (bytes.Buffer, strings.Builder)
  • Часто создаваемые структуры в hot path
  • Большие объекты (> 1 KB), которые переиспользуются

❌ НЕ используйте sync.Pool:

  • Для долгоживущих объектов
  • Для маленьких объектов (< 100 bytes)
  • Когда объектов создается мало

Продвинутый пример: HTTP handler с sync.Pool

package main

import (
    "encoding/json"
    "net/http"
    "sync"
)

type Response struct {
    Status string `json:"status"`
    Data   []byte `json:"data"`
}

var responsePool = sync.Pool{
    New: func() interface{} {
        return &Response{
            Data: make([]byte, 0, 1024),
        }
    },
}

func handler(w http.ResponseWriter, r *http.Request) {
    // Получить Response из пула
    resp := responsePool.Get().(*Response)
    defer responsePool.Put(resp)

    // Сброс состояния
    resp.Status = ""
    resp.Data = resp.Data[:0]

    // Заполнение
    resp.Status = "ok"
    resp.Data = append(resp.Data, []byte("hello world")...)

    // Отправка (JSON encoder сам аллоцирует буфер)
    w.Header().Set("Content-Type", "application/json")
    json.NewEncoder(w).Encode(resp)
}

func main() {
    http.HandleFunc("/", handler)
    http.ListenAndServe(":8080", nil)
}

Ограничения sync.Pool

⚠️ Важно:

  • Pool может быть очищен GC в любой момент
  • НЕ гарантируется, что объект останется в пуле
  • НЕ используйте для объектов с важным состоянием
  • Объекты должны быть безопасны для переиспользования
// ❌ ОПАСНО: состояние может быть потеряно
var connectionPool = sync.Pool{
    New: func() interface{} {
        conn, _ := sql.Open("postgres", "...")
        return conn // Плохо! Соединение может закрыться
    },
}

// ✅ ПРАВИЛЬНО: используйте специализированные пулы
// Для соединений: database/sql уже имеет пул
db, _ := sql.Open("postgres", "...")
db.SetMaxIdleConns(10)
db.SetMaxOpenConns(100)

package main

import (
    "encoding/json"
    "fmt"
    "sync"
)

type Message struct {
    ID      int    `json:"id"`
    Content string `json:"content"`
}

var messagePool = sync.Pool{
    New: func() interface{} {
        return &Message{}
    },
}

func goodJSONParsing(data []byte) {
    msg := messagePool.Get().(*Message)
    defer messagePool.Put(msg)

    msg.ID = 0
    msg.Content = ""

    fmt.Println(msg)
}

Пример 2: Мониторинг GC в production

Задача: Отслеживать метрики GC и экспортировать в Prometheus.

package main

import (
    "runtime"
    "time"

    "github.com/prometheus/client_golang/prometheus"
    "github.com/prometheus/client_golang/prometheus/promauto"
)

var (
    gcDuration = promauto.NewHistogram(prometheus.HistogramOpts{
        Name:    "go_gc_duration_seconds",
        Help:    "GC pause duration in seconds",
        Buckets: []float64{0.0001, 0.0005, 0.001, 0.005, 0.01, 0.05, 0.1},
    })

    heapAlloc = promauto.NewGauge(prometheus.GaugeOpts{
        Name: "go_heap_alloc_bytes",
        Help: "Current heap allocation in bytes",
    })

    gcCount = promauto.NewCounter(prometheus.CounterOpts{
        Name: "go_gc_total",
        Help: "Total number of GC cycles",
    })
)

func monitorGC() {
    var lastNumGC uint32

    ticker := time.NewTicker(10 * time.Second)
    defer ticker.Stop()

    for range ticker.C {
        var m runtime.MemStats
        runtime.ReadMemStats(&m)

        // Heap allocation
        heapAlloc.Set(float64(m.Alloc))

        // GC count
        if m.NumGC > lastNumGC {
            newGCs := m.NumGC - lastNumGC
            gcCount.Add(float64(newGCs))

            // Последняя пауза GC
            lastPause := m.PauseNs[(m.NumGC+255)%256]
            gcDuration.Observe(float64(lastPause) / 1e9) // В секунды

            lastNumGC = m.NumGC
        }
    }
}

func main() {
    go monitorGC()

    // Ваше приложение
    select {}
}

Пример 3: Оптимизация под контейнеры

Задача: Настроить GC для работы в Kubernetes pod с ограниченной памятью.

package main

import (
    "fmt"
    "os"
    "runtime"
    "runtime/debug"
    "strconv"
)

func setupGCForContainers() {
    // 1. Определить лимит памяти контейнера
    memLimitStr := os.Getenv("MEMORY_LIMIT") // Например, "2Gi"
    if memLimitStr != "" {
        memLimit := parseMemory(memLimitStr)

        // Установить GOMEMLIMIT = 90% от лимита контейнера
        softLimit := int64(float64(memLimit) * 0.9)
        debug.SetMemoryLimit(softLimit)

        fmt.Printf("Memory limit: %d MB (soft: %d MB)\n",
            memLimit/1024/1024, softLimit/1024/1024)
    }

    // 2. Настроить GOGC в зависимости от workload
    workload := os.Getenv("WORKLOAD_TYPE") // "cpu" или "memory"
    switch workload {
    case "cpu":
        // CPU-bound: реже GC, больше памяти
        debug.SetGCPercent(200)
    case "memory":
        // Memory-bound: чаще GC, меньше памяти
        debug.SetGCPercent(50)
    default:
        // По умолчанию
        debug.SetGCPercent(100)
    }

    // 3. Настроить GOMAXPROCS (обычно автоматически)
    // В контейнере может быть неправильно определен CPU count
    cpuLimit := os.Getenv("CPU_LIMIT") // Например, "2"
    if cpuLimit != "" {
        cpus, _ := strconv.Atoi(cpuLimit)
        runtime.GOMAXPROCS(cpus)
        fmt.Printf("GOMAXPROCS: %d\n", cpus)
    }
}

func parseMemory(s string) int64 {
    // Упрощенный парсинг "2Gi", "500Mi"
    // В production используйте библиотеку
    if len(s) < 2 {
        return 0
    }

    unit := s[len(s)-2:]
    value, _ := strconv.ParseInt(s[:len(s)-2], 10, 64)

    switch unit {
    case "Gi":
        return value * 1024 * 1024 * 1024
    case "Mi":
        return value * 1024 * 1024
    case "Ki":
        return value * 1024
    default:
        return value
    }
}

func main() {
    setupGCForContainers()

    // Вывод текущих настроек
    fmt.Printf("GOGC: %d%%\n", debug.SetGCPercent(-1))
    debug.SetGCPercent(100) // Восстановить

    var m runtime.MemStats
    runtime.ReadMemStats(&m)
    fmt.Printf("Initial heap: %d MB\n", m.Alloc/1024/1024)
}
