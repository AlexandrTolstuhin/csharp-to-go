# Инструкции для Claude по работе с курсом

## Общие принципы

### Язык
- Все материалы курса на **русском языке**
- Комментарии в коде на русском
- Commit messages на русском

### Стиль написания
- Ориентация на **C# разработчиков**
- Постоянное **сравнение C# ↔ Go**
- Акцент на **производительность и GC**
- **Практические примеры** с реальным кодом
- **Идиоматичный Go код** - показывать, как принято писать в Go-сообществе

### Структура материалов
- Каждый раздел начинается с **содержания**
- **Сравнительные таблицы** C# vs Go
- **Примеры кода** с подробными комментариями
- **Подсказки** (💡) и **предупреждения** (⚠️)
- Ссылки на следующий/предыдущий разделы

### Примеры кода
- Всегда показывать **C# код → Go код**
- Выделять **идиоматичные различия**
- Включать **полные рабочие примеры**
- Добавлять комментарии к неочевидным моментам
- Показывать **распространённые ошибки** и как их избежать

## Git workflow

### После завершения каждого подраздела

**ОБЯЗАТЕЛЬНО создавать коммит!**

```bash
# Проверить статус
git status

# Добавить новые файлы
git add .

# Создать коммит с понятным сообщением
git commit -m "Описание изменений"
```

### Формат commit messages

**Шаблон**:
```
[Раздел] Краткое описание

Детали:
- Что добавлено
- Что изменено
- Особенности
```

**Примеры**:
```bash
git commit -m "Завершена Часть 1: Основы Go

Добавлено:
- 03_key_differences.md (52 KB) - философия и идиомы Go vs C#
- 04_practice.md (35 KB) - 4 практических проекта

Обновлено:
- Навигация во всех файлах части 1
- .context-summary.md с прогрессом (16%)
"
```

```bash
git commit -m "Добавлен раздел 2.1: Горутины и каналы

Добавлено:
- Подробное объяснение горутин vs C# Task/Thread
- Каналы vs C# Channel<T>
- Паттерны работы с каналами
- 15+ примеров кода
"
```

### Когда делать коммит

✅ **Делать коммит**:
- После завершения каждого файла .md
- После завершения логической части (например, всей Части 1)
- После обновления навигации
- После добавления практических проектов
- После исправления ошибок в существующих материалах
- После дополнения существующих разделов (если были изменения в Go или появилась новая информация)

❌ **Не коммитить**:
- Незавершённые файлы (кроме как с пометкой WIP)
- Файлы с заглушками
- Временные файлы

### Push в удалённый репозиторий

**ОБЯЗАТЕЛЬНО делать push после завершения подраздела!**

```bash
# После создания коммита(ов)
git push origin master
```

✅ **Когда делать push**:
- После завершения каждого подраздела (например, 2.4 Примитивы синхронизации)
- После завершения целой части (например, Часть 1)
- После важных обновлений (добавление больших разделов, исправления)
- В конце рабочей сессии

💡 **Совет**: Лучше делать push чаще, чтобы не потерять работу.

## Обновление файлов проекта

### После каждого значительного изменения обновлять:

#### 1. `.context-summary.md` (ОБЯЗАТЕЛЬНО)

Структура проекта, прогресс и дата обновляются **автоматически** скриптом:

```bash
python update_context.py
```

Скрипт перегенерирует секции `<!-- AUTO: STRUCTURE -->` и `<!-- AUTO: PROGRESS -->`, не трогая ручные части файла.

##### Когда запускать

✅ **Запускать обязательно:**
- Добавлен новый файл `.md` в любую часть курса
- Завершён раздел (файл дорос до финального размера)
- Добавлена новая папка `project*` в `part5-projects/`
- Перед каждым коммитом, если менялась структура файлов

❌ **Не нужно запускать:**
- При правке содержимого существующего файла (размер не меняется значительно)
- При правке якорей, заголовков внутри файла

##### Что скрипт обновляет автоматически
- Дерево файлов с реальными размерами в KB и статусами ✅/🔲
- Прогресс по каждой части и общий прогресс курса
- Дату последнего обновления
- Версию проекта (при флаге `--version`)

##### Что обновлять вручную
Только разделы, которые существуют в файле прямо сейчас:
- `### Репозиторий` — если изменился путь, ветка и т.п.
- `### Особенности курса` — если добавились принципиально новые особенности
- `### Технические детали` — если изменилась техническая конфигурация

##### Строгие правила работы с .context-summary.md

⛔ **ЗАПРЕЩЕНО:**
- Редактировать содержимое внутри `<!-- AUTO: ... -->` / `<!-- /AUTO: ... -->` блоков вручную — они перезапишутся скриптом
- Воссоздавать разделы, которых нет в файле (`### Текущая работа`, `### История актуализации`, `### Следующий шаг` — они были намеренно удалены)
- Добавлять новые разделы в файл без явной просьбы пользователя

✅ **Разрешено:**
- Запустить `python update_context.py` для обновления авто-секций
- Редактировать существующие ручные разделы (`### Репозиторий`, `### Особенности курса`, `### Технические детали`)

##### Обновление версии проекта

`**Версия**` автоматически обновляется во всех файлах через флаг `--version`:

```bash
python update_context.py --version "0.9.0 (описание изменений)"
```

Скрипт находит все `.md` файлы с маркером `<!-- AUTO: VERSION -->` и заменяет строку версии.

**Когда менять версию** (semver: MAJOR.MINOR.PATCH):
- `PATCH` (0.8.0 → 0.8.1): исправления, дополнения к существующим разделам
- `MINOR` (0.8.0 → 0.9.0): добавлен новый раздел или завершена часть курса
- `MAJOR` (0.8.0 → 1.0.0): курс полностью завершён

**Как определить версию**: прочитать текущую `**Версия**` из `.context-summary.md` (между `<!-- AUTO: VERSION -->` маркерами) и применить semver-логику выше. Описание — краткое: что изменилось.

#### 2. `README.md` (корневой - ОБЯЗАТЕЛЬНО!)
⚠️ **ВАЖНО**: Это файл часто забывают обновлять!

Что обновлять:
- **Прогресс-бар**: `[![Progress](https://img.shields.io/badge/Progress-XX%25-orange)]`
- **Статусы разделов**:
  - `_(в разработке)_` → убрать после завершения
  - Добавить ✅ к завершённым пунктам
  - Добавить ссылки на файлы
- **Описание разделов**: раскрыть подпункты для завершённых разделов

Пример обновления:
```markdown
# Было:
#### 1.3 Ключевые отличия от C# _(в разработке)_
- Отсутствие классов и наследования

# Стало:
#### 1.3 [Ключевые отличия от C#](./part1-basics/03_key_differences.md)
- ✅ Отсутствие классов и наследования
```

- Процент прогресса части
- Описание разделов
- Навигация

## Процесс написания статей

### Инкрементальное написание (ВАЖНО!)

**Статьи нужно писать небольшими частями**, а не генерировать весь файл целиком за один раз.

#### Почему это важно:
- Большие файлы сложнее генерировать без ошибок
- Легче отслеживать прогресс
- Проще вносить коррективы по ходу работы
- Меньше риск потери контента при сбоях

#### Процесс написания:

1. **Создать план раздела** (оглавление)
   ```markdown
   # X.Y Название раздела

   ## Содержание
   - [Подраздел 1](#подраздел-1)
   - [Подраздел 2](#подраздел-2)
   ...
   ```

2. **Писать по одному подразделу за раз**
   - Написать подраздел 1 → сохранить
   - Написать подраздел 2 → сохранить
   - И так далее...

3. **После каждого подраздела**:
   - Проверить корректность
   - При необходимости — промежуточный коммит (WIP)

4. **В конце — финализация**:
   - Проверить весь файл целиком
   - Добавить навигацию
   - Создать финальный коммит

#### Пример workflow:

```
Шаг 1: Создать файл с содержанием и введением
Шаг 2: Написать "Основные концепции"
Шаг 3: Написать "Сравнение с C#"
Шаг 4: Написать "Практические примеры"
Шаг 5: Добавить навигацию
Шаг 6: Финальная проверка и коммит
```

#### Размер частей:
- Оптимальный размер одной части: **500-1500 строк**
- Максимум за одну итерацию: **2000 строк**
- Если раздел большой — разбить на несколько файлов или частей

## Автогенерация содержания (doctoc)

### Инструменты

Для работы с содержанием есть два инструмента:

**`add_doctoc_markers.py`** — добавляет doctoc-маркеры в новые файлы и запускает doctoc:

```bash
python add_doctoc_markers.py                    # весь проект
python add_doctoc_markers.py part2-advanced/    # конкретная папка
python add_doctoc_markers.py part1-basics/02_syntax_basics.md  # файл
```

**`npx doctoc`** — только обновить TOC (маркеры уже есть):

```bash
npx doctoc part1-basics/02_syntax_basics.md --github --notitle
```

### Когда запускать

**ОБЯЗАТЕЛЬНО запускать после любых изменений заголовков** (`##`, `###`, `####`):
- Добавлены новые заголовки
- Переименованы существующие заголовки
- Изменён порядок или удалены заголовки

```bash
# Стандартный сценарий: изменили заголовки в файле → обновить TOC
npx doctoc <файл>.md --github --notitle

# Новый файл без маркеров → добавить маркеры и сгенерировать TOC
python add_doctoc_markers.py <файл или папка>
```

## Особенности контента

### Акценты на идиоматичности

При написании материалов **обязательно** показывать:

1. **Не только "как сделать", но и "как принято"**
   ```go
   // ❌ Не идиоматично (работает, но не Go-way)
   func GetUser(id int) (*User, error) { ... }

   // ✅ Идиоматично
   func getUser(id int) (*User, error) { ... }
   ```

2. **Распространённые ошибки C# разработчиков**
   - Попытка использовать наследование
   - Игнорирование ошибок
   - Неправильная работа с nil
   - Излишнее использование интерфейсов
   - Создание getter/setter для всего

3. **Паттерны миграции**
   - Как заменить LINQ на циклы
   - Как заменить async/await на горутины
   - Как заменить DI на простую композицию
   - Как заменить exceptions на errors

### Актуализация материалов

**По возможности дополнять существующие разделы**, если:

1. **Появилась новая информация** о теме:
   - Новая версия Go с изменениями в обсуждаемой области
   - Новые best practices
   - Исправления неточностей

2. **Обнаружены пробелы** в материале:
   - Пропущенные важные аспекты
   - Недостаточно примеров
   - Нужны дополнительные пояснения

3. **Связанные темы**:
   - Если при работе над новым разделом выявилась необходимость дополнить предыдущий
   - Добавление ссылок между связанными разделами

**Примеры**:
- При работе над разделом 2.4 (Примитивы синхронизации) обнаружено, что не хватает раздела про `golang.org/x/sync` → **дополнить раздел**
- Вышла новая версия Go с изменениями в GC → **обновить раздел 2.3**
- Обнаружена неточность в примере → **исправить и создать коммит**

⚠️ **Важно**: После дополнения создать отдельный коммит с описанием изменений.

### Производительность

При обсуждении производительности включать:
- **Benchmarks** с конкретными цифрами
- **Memory allocations**
- **GC pressure**
- Сравнение с C# (когда уместно)
- **pprof** примеры

### Практические проекты

Каждый практический проект должен:
- Решать **реальную задачу**
- Иметь **поэтапную реализацию**
- Включать **полное решение** в `<details>`
- Предлагать **дополнительные задания**
- Показывать **тестирование**
