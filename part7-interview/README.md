## О чём эта часть

Лайфкодинг — стандартная часть технического интервью в российских компаниях. Задачки пишутся в реальном времени: в онлайн-редакторе (CoderPad, leetcode.com) или на доске. Интервьюер смотрит не только на итоговый код, но и на **процесс мышления**, умение задавать уточняющие вопросы и обосновывать решения.

### Что ожидают от Go-разработчика

| Уровень | Ожидания |

## Как пользоваться этой частью

### Подход к решению задач на интервью

1. **Уточни условие** — задай 2-3 вопроса до написания кода
   - "Входные данные могут быть nil?"
   - "Нужна thread-safety?"
   - "Какой порядок важен?"

2. **Проговори подход** — скажи вслух что собираешься делать и почему

3. **Напиши код** — начни с рабочего наивного решения

4. **Проверь граничные случаи** — пустой ввод, один элемент, дубликаты

5. **Оцени сложность** — Big O по времени и памяти

6. **Предложи оптимизацию** — даже если не реализуешь

### Типичные ошибки C# разработчиков на Go-интервью

```go
// ❌ Пишут как в C# — создают лишние объекты
func reverse(s []int) []int {
    result := make([]int, len(s))
    for i, v := range s {
        result[len(s)-1-i] = v
    }
    return result
}

// ✅ Идиоматично Go — переворачиваем in-place
func reverse(s []int) {
    for i, j := 0, len(s)-1; i < j; i, j = i+1, j-1 {
        s[i], s[j] = s[j], s[i]
    }
}
```

```go
// ❌ Используют sync.Mutex там, где хватает канала
type Counter struct {
    mu    sync.Mutex
    count int
}

// ✅ Или наоборот — городят канал там, где нужен мьютекс
// Правило: мьютекс для защиты данных, канал для коммуникации
```

## Формат каждой задачи

Каждая задача в этой части содержит:

- **Компания / источник** — где встречается задача
- **Уровень** — Junior / Middle / Senior
- **Формулировка** — точно как на интервью
- **Что проверяют** — цель задачи с точки зрения интервьюера
- **Разбор** — анализ подхода
- **Решение** — идиоматичный Go код
- **Типичные ошибки** — что делают неправильно
- **Дополнительные вопросы** — что могут спросить следом
